# Target V2 语流组与全局停顿分配设计

## 1. 文档目的

本文描述 `翻译合成` 下一阶段的原理级改造方案。

目标不是修补当前某几个问题段，而是从时序分配模型本身解决：

1. 合成段之间出现过大的静音间隔
2. 各段语速不一致
3. 听感上连贯性明显弱于原语音

本文不是全量待实现。

当前状态：

1. 第一步“全局停顿预算分配”已经落地
2. 第二步“语流组建模”尚未开始
3. 第三步“组级 TTS”尚未开始

## 2. 当前实现的问题根因

当前代码在 [pipeline.py](/Volumes/8TR0/codex/video_translater/backend/app/pipeline.py) 中采用的是：

1. 逐段生成 `target draft`
2. 为每段规划一个最终时间窗
3. 通过 `slot_audio(...)` 把每段音频塞进该时间窗
4. 在 `_build_target_track(...)` 中，把段与段之间的空档直接用 `make_silence(...)` 补齐

这会产生一个结构性问题：

- 任何未被语音消费掉的时间，最终都会表现为“绝对静音”
- 这些静音不区分“自然停顿”还是“对齐残余”
- 因此听感会出现：
  - 某些地方停顿过长
  - 有些语句衔接生硬
  - 整体节奏比原语音更碎

一句话：

当前系统把“全局富余时间”错误地转换成了“局部段间静音”。

## 3. V2 设计目标

V2 需要同时满足以下条件：

1. 整条目标音轨与原音轨起始时间、结束时间严格对齐
2. 每个 target 段的最终时间段，必须与对应 source 段时间段有交集
3. 相邻段之间的停顿应尽量接近原语音的停顿结构
4. 各段语速应尽量一致，不应让少数段明显被压快或拖慢
5. 不再把剩余时间简单变成纯静音

## 4. 核心思路

V2 的核心不是“逐段塞回去”，而是：

`先识别自然语流组 -> 再做全局时间分配 -> 把停顿预算只投放到合理边界 -> 最后回写到字幕段`

也就是说，V2 的基本单位不再只是“字幕段”，而是分两层：

1. `segment`
用户界面、编辑、纠错、保存仍然以字幕段为单位

2. `utterance group`
时序规划和连贯性优化以语流组为单位

## 5. 关键概念

### 5.1 Segment

当前 `source/target` 文本框里的单条字幕。

### 5.2 Boundary

相邻两个段之间的边界。

每个边界都应被分类，例如：

1. `tight`
几乎无停顿，应连续衔接

2. `soft_pause`
允许短暂停顿

3. `hard_pause`
允许明显停顿

### 5.3 Utterance Group

一组应连续说完的相邻字幕段。

组内不应出现明显人工静音。
组与组之间才允许分配较大的停顿。

## 6. 语流组生成规则

### 6.1 输入

语流组生成依赖以下输入：

1. source 段时间轴
2. source 段文本
3. source 段之间的原始 gap
4. target draft 文本长度与 target draft 音频时长

### 6.2 分组规则

相邻两个段，如果满足以下任意条件，应优先归入同一组：

1. 原始 source gap 很短
建议阈值：`< 180ms`

2. 前段文本在语义上未闭合
例如逗号、列举、递进、自问自答、承接句

3. target draft 自身读起来明显连续
例如英文里一句话被拆成了多个中文字幕段

相邻两个段，如果满足以下条件，应优先断组：

1. 原始 source gap 较长
建议阈值：`> 450ms`

2. 前段有完整句号或强终止语气

3. 原音里存在清晰换气、停顿、转折

### 6.3 结果

输出一份组计划，例如：

- `group-0001 = seg-0001 ~ seg-0003`
- `group-0002 = seg-0004 ~ seg-0006`

建议新增产物：

- `segments/target.group-plan.v1.json`

## 7. 时间预算模型

### 7.1 不是只分配段时长

V2 需要联合优化两类变量：

1. 每段目标时长
2. 每个边界的停顿时长

当前 V1 只是在规划段窗口，实际上没有把“边界停顿”作为显式变量处理。

### 7.2 总时间约束

总约束保持不变：

1. 第一段开始时间与总音轨开始对齐
2. 最后一段结束时间与总音轨结束对齐

### 7.3 段交集约束

每个段仍然必须满足：

`target_window_i ∩ source_window_i != empty`

即每个目标段最终时间窗必须和原始对应段有交集。

### 7.4 优化目标

V2 的优化目标建议为：

1. 最小化整体语速缩放方差
不要让某一两段明显更快或更慢

2. 最小化“非自然停顿”总量
尤其要惩罚出现在 `tight boundary` 上的大停顿

3. 最大化停顿与原始停顿结构的一致性
长停顿优先分配给 `hard_pause`

4. 最小化段间纯静音片段数量

### 7.5 预算分配原则

如果整条 target draft 比 source 更短，产生了 slack：

1. 先用于降低各段语速差异
2. 再分配给 `hard_pause`
3. 再分配给 `soft_pause`
4. 严禁把大量 slack 直接投放到 `tight boundary`

## 8. 两阶段时序规划

### 阶段 A：组级规划

先以 `utterance group` 为单位规划整组的开始、结束和停顿预算。

这一步只决定：

1. 哪些段连续说
2. 哪些地方允许真正停顿
3. 每个组大概可以占多少时间

### 阶段 B：组内分配

再在组内部把时间分配回每个 segment。

组内目标是：

1. 连续发声
2. 小停顿尽可能少
3. 各 segment 的语速尽量接近组内统一语速

这样就不会再出现“整条时间轴很宽松，但局部段间却被硬塞出大静音”的情况。

## 9. 合成策略升级

### 9.1 V2 推荐主方案

从长期看，最佳方案是：

`按语流组合成，而不是按字幕段独立合成`

也就是说：

1. 文本仍逐段翻译与逐段展示
2. 但 TTS 合成单位提升为 `utterance group`

优点：

1. 组内天然连续，不会每段都重新起声
2. 组内不再需要额外静音拼接
3. 听感更接近自然说话

### 9.2 UI 不需要改成“按组合显示”

前端仍然按 segment 展示。
只是在后台记录：

1. 某个 segment 属于哪个 group
2. 它在 group 音频里的相对时间范围

这样就可以保持现有的：

- 逐段编辑
- 逐段预览
- 左右滚动同步

## 10. 文件产物建议

建议新增以下中间产物：

1. `segments/target.group-plan.v1.json`
记录：
- group id
- 包含哪些 segment
- 边界类型
- 组级时间预算

2. `voices/target-groups/group-xxxx.wav`
组级合成音频

3. `segments/target.segment-timing.v2.json`
记录每个 segment 在组音频中的相对时间和最终全局时间

## 11. 与当前 V1 的关系

### 11.1 保留项

以下能力可以保留：

1. source snapshot
2. target 文本逐段编辑
3. `重新合成`
4. 目标视频渲染
5. 右侧逐段可播放预览

### 11.2 替换项

以下实现应逐步替换：

1. 逐段对齐窗口规划
2. `_build_target_track(...)` 中“空档直接补静音”的逻辑
3. 段级独立 TTS 作为唯一合成方式

## 12. 实施顺序

建议分三步落地：

### 第一步：停顿预算显式化

当前状态：已完成。

当前代码已经完成：

1. boundary 分类
2. 全局停顿预算分配
3. 统一语速放宽

先保留当前逐段 TTS，不动参考音频策略。

只改：

1. 增加 boundary 分类
2. 增加全局停顿预算分配
3. 不再把 slack 直接全部变成 silence

这一步已经可以验证“长空档是否明显减少”。

### 第二步：引入语流组

当前状态：下一阶段待开始。

增加 `utterance group` 计划文件和组级时间预算。

这一步开始按组规划，而不是单纯按 segment 规划。

### 第三步：升级为组级 TTS

当前状态：未开始。

最终把合成单位从 segment 提升到 group。

这是连贯性最强的最终形态。

## 13. 验收标准

实施完成后，应满足：

1. 总时长严格等于原音频总时长
2. 每个 target segment 都与对应 source segment 有交集
3. 段间大静音显著减少
4. 连续说话区域的停顿听感接近原音频
5. 各段语速差异显著减小
6. 前端逐段预览和滚动联动能力不受影响

## 14. 一句话总结

V2 的本质不是“更聪明地补静音”，而是：

`把时间分配从“逐段硬塞 + 空档补静音”，升级成“语流组建模 + 全局停顿预算分配 + 组内连续说话”`
