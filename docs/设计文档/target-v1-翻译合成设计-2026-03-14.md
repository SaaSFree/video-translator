# Target V1 翻译合成设计

## 1. 文档目的

本文描述当前代码里已经落地的 `翻译合成` 实现。

它不是未来规划，而是当前真实存在的行为说明，覆盖：

1. 进入条件
2. 逐段翻译与逐段合成
3. `重新合成`
4. 全局时序对齐
5. target 视频输出

## 2. 当前实现目标

当前 `翻译合成` 已经能够稳定产出：

1. `target draft` 文本和分段音频
2. `target aligned` 最终时间轴与分段音频
3. `target-track.v1.wav`
4. `target/dubbed.v1.mp4`
5. 右侧逐段预览与最终视频预览

## 3. 进入条件

### 3.1 直接点击 `翻译合成`

必须满足：

1. 已有 `segments/source.v1.json`
2. 已有 `source/original.wav`
3. `source review` 没有待确认项

如果还有待确认项，前端会阻止继续执行。

### 3.2 点击 `转写合成`

当前代码行为：

1. 先跑 source 链路
2. 若 source review 还有待确认项，则自动接受全部修订
3. 继续跑 target

所以：

- `翻译合成` 对人工确认是严格门槛
- `转写合成` 会自动越过这道门

## 4. 输入

当前 target 链路输入为：

- `segments/source.v1.json`
- `source/original.mp4`
- `source/original.wav`
- `target_language`
- 当前系统 `review_backend`
- 当前系统 `tts_model`

### 4.1 Source 快照

每次进入 target 链路时，会先构建并冻结一份：

- `segments/source.snapshot.for-target.v1.json`

后续 target 全部基于这份 snapshot，而不是处理中途再去读会变化的 source。

### 4.2 参考音频

当前 target TTS 的参考输入使用的是与 source 段严格对应的 source segment 音频，而不是临时从整条音频再裁 prompt。

也就是说：

- `ref_audio` 来自 `voices/source-segments/<segment_id>.wav`
- `ref_text` 来自 snapshot 里的同段 source 文本

## 5. 模型配置

### 5.1 文本校译 / 翻译

使用 `review_backend`，当前可选：

- `codex-none`
- `codex-low`
- `codex-medium`
- `codex-high`
- `codex-xhigh`

### 5.2 语音合成

使用 `tts_model`，当前可选：

- `mlx-community/Qwen3-TTS-12Hz-0.6B-Base-8bit`
- `mlx-community/Qwen3-TTS-12Hz-1.7B-Base-8bit`

### 5.3 运行时服务切换

这是当前代码中的关键行为：

如果 TTS 或 ASR 端口上已经挂着旧模型，而下拉设置改成了新模型，后端会：

1. 先停掉旧服务
2. 再按当前设置启动新服务
3. 然后继续执行

不会再像旧版本那样直接因为模型不匹配而整轮失败。

## 6. 总流程

当前 `翻译合成` 实际执行：

```text
读取 source 最终文本
  -> 冻结 source snapshot
  -> 清空旧 target 产物
  -> 逐段翻译
  -> 逐段合成 target draft
  -> 每完成一段就写 draft 并在右侧显示
  -> 全部完成后做全局时序对齐
  -> 生成 target track
  -> 渲染 dubbed.v1.mp4
```

## 7. 逐段翻译与逐段合成

### 7.1 逐段翻译

当前 target 文本由 `CodexTargetTranslator` 逐段生成。

特点：

1. 一段一段独立翻译
2. 提示词会要求输出“最终朗读文本”
3. 数字、百分比、货币、日期、时间、缩写等会被转换成适合朗读的文字形式

### 7.2 逐段合成

每个 source 段会生成：

- `voices/target-draft/<segment_id>.wav`

同时写入：

- `segments/target.draft.v1.json`

每完成一段就会：

1. 右侧条目立即显示
2. 右侧可逐段试听
3. 状态栏 `x/y` 实时推进

## 8. Target 条目与占位

在 target 还没全部生成完时，右侧不会只显示已经完成的少量条目，而是：

1. 以左侧 source 条目数为骨架
2. 右侧同步生成相同数量的 target 条目
3. 未完成条目显示为空文本与 `待生成`

这样可以保证：

- 用户在生成过程中滚动页面时
- 左右条目仍能完全对齐

## 9. Target 编辑与重新合成

### 9.1 编辑

右侧 `target` 文本可以直接编辑。

保存按钮规则：

- 未改动时禁用
- 有真实改动时启用

### 9.2 重新合成

当 target 文本存在已保存改动时，右侧主按钮切换为：

- `重新合成`

点击后走 `target-resynthesize` 路径，只做：

1. 读取现有 target 文本
2. 重新合成语音
3. 重新全局对齐
4. 重新拼接 target track
5. 重新生成 dubbed 视频

不会重新翻译文本。

## 10. 全局时序对齐

### 10.1 原则

当前实现已经不是“每段硬塞回原段时长”，而是：

1. 先拿到每段 draft 的真实时长
2. 再在整条时间线上统一规划所有段的最终时间窗
3. 保证整条轨道首尾与 source 总时长严格对齐
4. 保证每个 target 段与对应 source 段有交集

### 10.2 求解方式

当前代码使用的是“有界单调全局投影”思路：

- 先为每段给出偏好的居中时间
- 再在全局时间线上做单调、带上下界的统一求解
- 最后得到每段 `planned_start / planned_end`

这样能把长停顿和富余时隙当成全局 slack 使用，而不是只在局部拥挤段里硬挤。

### 10.3 对齐输出

对齐完成后写出：

- `voices/target-aligned/<segment_id>.wav`
- `segments/target.aligned.v1.json`
- `voices/target-track.v1.wav`

## 11. 最终视频输出

target 结束后会把：

- `source/original.mp4`
- `voices/target-track.v1.wav`

合成为：

- `target/dubbed.v1.mp4`

右侧视频区优先挂这个文件。

若还没生成完成，则回退显示 source 视频画面，同时用 target 分段音频做联动试听。

## 12. 关键产物

当前 target 链路核心文件如下：

- `segments/source.snapshot.for-target.v1.json`
- `segments/target.draft.v1.json`
- `segments/target.aligned.v1.json`
- `subtitles/target.draft.v1.srt`
- `subtitles/target.v1.srt`
- `voices/target-draft/*.wav`
- `voices/target-aligned/*.wav`
- `voices/target-track.v1.wav`
- `target/dubbed.v1.mp4`

## 13. 状态流

当前 target 阶段常见状态：

- `translating_target`
- `synthesizing_target`
- `aligning_target`
- `rendering_target_video`
- `complete`
- `error`

对应前端摘要通常显示为：

- `正在翻译合成（x/y）`
- `正在重新合成（x/y）`
- `正在时序对齐（x/y）`
- `翻译合成已完成`

## 14. 当前已知边界

当前 target 链路仍然有这些已知边界：

1. 逐段独立翻译仍可能造成少量跨段承接不自然
2. 段间轻微杂音仍可能存在
3. 当前参考音频直接来自按字幕硬切的 `source-segments`，尚未升级成专门的“声学参考清单”
4. 逐段独立 TTS 仍可能把参考音频边界气口或噪声门瞬态重复带入段首

也就是说，当前 target 已经可用，但音质和边界自然度仍有继续优化空间。
